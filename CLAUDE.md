# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Personal dotfiles + multi-host bootstrap. Two entry points:

- **`install.sh`** ‚Äî run locally to symlink configs into `$HOME` and install CLI tools (glow, nvim, node, uv, Claude Code, Codex, VS Code `code` tunnel CLI, etc.). The shell aliases also define a `vscode-tunnel` helper ‚Äî see `shell/bashrc_aliases`.
- **`deploy.sh`** ‚Äî run locally to bootstrap a remote server over SSH (authorize SSH key, copy `gh`/Claude/Codex auth, clone this repo on the remote, then invoke `install.sh` there).

Shared helpers (`run_step`, `retry`, `backup_and_link`, `backup_and_copy`) live in `lib/common.sh` and are sourced by both entry points plus `scripts/install_claude_plugins.sh`.

## Architecture

- **Topical layout**: configs grouped under `shell/`, `editor/`, `tmux/`, `git/`, `ai/` (with `ai/skills/<name>/SKILL.md` for Claude Code skills), `chpc/` (the Notchpeak HPC agent guide deployed to `~/CLAUDE.md`, CHPC only), `cloudlab/` (the CloudLab bare-metal agent guide deployed to `~/CLAUDE.md`, CloudLab only). Operational scripts under `scripts/`. Shared shell library under `lib/`. The three top-level scripts (`install.sh`, `deploy.sh`, `uninstall.sh`) plus docs are the only flat files. No `src/` or runtime-tests tree ‚Äî regression tests live at `scripts/test_regressions.sh`.
- **Error handling**: scripts use `set -uo pipefail` (no `-e`). Failures are tracked per-step via `run_step()` into a `FAILURES=()` array. The script continues and reports all failures at the end.
- **Paths**: all paths inside scripts are relative to `$DIR` (derived via `$(cd "$(dirname "$0")" && pwd)`). Never hardcode absolute paths.
- **Generated compat configs**: `~/.dotfiles-generated/` holds version-adaptive variants regenerated by `install.sh` on every run:
  - `tmux.compat.conf` ‚Äî terminal type, true color, passthrough
  - `tmux-theme.conf` + `tmux-theme-light.conf` + `tmux-theme-dark.conf` ‚Äî status/pane palette dispatcher and per-mode palettes (uses `*-style` on tmux ‚â• 2.9, legacy `*-bg`/`*-fg` on older). `~/.tmux-theme.conf` symlinks to the dispatcher.
  - `vimrc.compat` ‚Äî clipboard, listchars, Neovim detection
  - `gitconfig.compat` ‚Äî credential helper; on keyless hosts also rewrites `git@github.com:` URLs to HTTPS so they ride the helper
  - `bashrc_compat` ‚Äî tool-specific aliases with version-appropriate flags
  - **Never edit generated files directly** ‚Äî edit the `render_*()` functions in `install.sh`.

## Conventions

- **Symlinks** for most configs via `backup_and_link(src, dst)`. **Copies** for secrets/tool configs via `backup_and_copy(src, dst)` (allows local overrides without affecting the repo).
- Existing files are backed up as `.bak` before replacing. `uninstall.sh` restores them.
- **Idempotency**: every install function checks if the tool exists before installing. Use `--force` to reinstall. Re-running `install.sh` is always safe.
- **Architecture detection**: install functions handle `x86_64` and `aarch64`. Unsupported architectures skip gracefully.
- `retry()` wraps network calls (3 attempts, 2-second delay).

```
  ‚îî‚îÄ‚îÄ sources ~/.zshrc_exports  (symlink ‚Üí shell/zshrc_exports)
       ‚îî‚îÄ‚îÄ sources ~/.dotfiles-generated/bashrc_compat  (POSIX-clean, shared)
  ‚îî‚îÄ‚îÄ sources ~/.zshrc_aliases  (symlink ‚Üí shell/zshrc_aliases)
       ‚îî‚îÄ‚îÄ sources ~/.dotfiles-generated/bashrc_compat

~/.vimrc  (symlink ‚Üí editor/vimrc)
  ‚îî‚îÄ‚îÄ sources ~/.dotfiles-generated/vimrc.compat

~/.tmux.conf  (symlink ‚Üí tmux/tmux.conf)
  ‚îî‚îÄ‚îÄ sources ~/.dotfiles-generated/tmux.compat.conf

~/.gitconfig  (symlink ‚Üí git/gitconfig)
  ‚îî‚îÄ‚îÄ includes ~/.dotfiles-generated/gitconfig.compat

~/.claude/settings.json   (copy of ai/claude_settings.json)
~/.codex/config.toml      (copy of ai/codex_config.toml)

~/CLAUDE.md  (symlink ‚Üí chpc/CLAUDE.md on CHPC, or cloudlab/CLAUDE.md on CloudLab;
             mutually exclusive, gated by is_chpc() / is_cloudlab())

~/.claude/skills/<name>   (symlink ‚Üí ai/skills/<name>, one per subdir)
                          (also: symlink ‚Üí ~/.local/share/claude-skills/<repo>/skills/<name>
                                 for upstream skills cloned by
                                 scripts/install_claude_skills.sh)
~/.local/share/claude-skills/   (clone cache for obra/superpowers and
                                 anthropics/skills; refreshed on install
                                 --force; removable via uninstall.sh)
```

**Skills vs. settings**: `ai/claude_settings.json` and `ai/codex_config.toml`
are **copies** (per-host overrides expected). Each `ai/skills/<name>/` is a
**symlink** ‚Äî skills are shared knowledge that should track the repo
verbatim, like `editor/nvim` or `shell/bashrc_aliases`. Upstream skills
follow the same symlink pattern, but their content lives in a clone cache
at `~/.local/share/claude-skills/` rather than in this repo. See
[`docs/ai-skills.md`](docs/ai-skills.md) for the bundled set, the upstream
curated subset (7 obra + 4 anthropic-markdown + `pdf`), and authoring notes.
The marketplace-plugin half of `scripts/install_claude_plugins.sh` installs
`context7`, `commit-commands`, and `pr-review-toolkit`; the MCP half
registers one server (`fetch`).

Bash and zsh have **parallel rc files** under `shell/` (`bashrc_exports`/`bashrc_aliases` ‚Üî `zshrc_exports`/`zshrc_aliases`). Keep behavior in sync when editing either side ‚Äî the aliases are nearly identical, the exports diverge on prompt, hooks (`PROMPT_COMMAND` ‚Üî `precmd`), shell options (`shopt` ‚Üî `setopt`), and tool init flags (`init bash` ‚Üî `init zsh`). The generated `bashrc_compat` file is POSIX-clean and sourced unchanged by both shells. Zsh wiring runs on macOS unconditionally (default login shell) and on Linux hosts only when `zsh` is installed.

## Setup

```bash
./install.sh               # idempotent ‚Äî re-run is safe
./install.sh --force       # reinstall CLI tools even if present
./install.sh --use-modules # on CHPC, prefer module load over binary install
```

## Remote deploy

```bash
./deploy.sh              # interactive: pick steps, enter host
./deploy.sh --yes        # accept defaults, skip confirmations
```

Claude and Codex copy only known auth files (`~/.claude/.credentials.json`, `~/.codex/auth.json`) from the local machine to the remote. GitHub CLI auth is recreated from the local `gh auth token` before cloning, with `hosts.yml` used only as a plaintext-token fallback. Keychain-backed Claude auth is reported as non-transferable and must be set up on the remote with `claude auth login` or `claude setup-token`. Auth material is **never** committed to this repo (see `.gitignore`).

## Vim & Neovim

- **editor/vimrc** is plugin-free (uses only built-in vim features + netrw). Don't add plugin managers or plugin dependencies.
- **editor/nvim/** is the Neovim config (symlinked to `~/.config/nvim/`). Uses lazy.nvim for plugins.
  - Layout: `init.lua` ‚Üí `lua/config/{options,keymaps,autocmds,lazy,compile_commands}.lua` ‚Üí `lua/plugins/*.lua`
  - Leader is Space.
  - 15 plugin spec files in `lua/plugins/`. Each file = one plugin or group.
  - **LSP**: pyright (tuned to `openFilesOnly`) + ruff (lint only, hover disabled) + clangd (`--background-index --clang-tidy`) + lua_ls. `<C-]>`/`F12` jump to definition, `<F2>` renames, `<leader>uh` toggles inlay hints. See `docs/neovim.md` for the full table.
  - **clangd auto-bootstrap**: on first attach, `config/compile_commands.lua` runs `cmake -B build -DCMAKE_EXPORT_COMPILE_COMMANDS=ON` for CMake projects and symlinks the result to the root. Make projects expose `:GenCompileCommands` (needs `bear`).
  - Which-key groups: `<leader>f` = find, `<leader>g` = git, `<leader>b` = buffer.
  - **Oil replaces netrw** (`vim.g.loaded_netrw = 1`). `<leader>e` and `-` open Oil.
  - **`<leader>m` differs between vim and nvim**: vim runs `!glow %` (terminal); nvim triggers `MarkdownPreviewToggle` (browser) on markdown files.
  - **`editor/nvim/lazy-lock.json` is gitignored** ‚Äî each host manages its own lock so `:Lazy update` doesn't dirty the repo working tree. Plugin versions are not pinned across hosts (personal dotfiles, not reproducible builds). Run `:Lazy sync` after editing plugin specs.
  - Setup installs nvim via AppImage (skipped on glibc < 2.28). Pass `--use-modules` on CHPC to prefer `module load` instead.
- **tmux prefix** is `Ctrl-a` (not the default `Ctrl-b`).
- **Indentation**: vimrc defaults to 4-space tabs; web filetypes (html/css/js/ts/json/yaml) use 2-space via autocmd. Neovim adds `lua` to the 2-space list.
- Glow is used for markdown preview in vim (`<leader>m`).
- **Theme override**: shell exposes `theme light|dark|auto` (shell/bashrc_aliases / shell/zshrc_aliases). It sets `DOTFILES_THEME`, propagates to tmux via `set-environment -g`, and re-sources `~/.tmux-theme.conf`. Running nvim/vim instances need `:set background=light|dark` to repaint (handled by `colorscheme.lua`'s `OptionSet` autocmd). Detection covers VS Code (local + Remote-SSH via `~/.vscode-server/.../storage.json`), Apple Terminal, OSC 11 (outside tmux), and `COLORFGBG`; falls back to dark.

## CHPC (Center for High Performance Computing)

`is_chpc()` in `lib/common.sh` detects CHPC systems via hostname (`*.chpc.utah.edu`) or path (`/uufs/chpc.utah.edu`). When on CHPC, `install.sh` automatically:

- Installs gh, Node.js, uv, btop, Neovim, Claude Code, and Codex via direct binary download (same as non-CHPC)
- Pass `--use-modules` (or `CHPC_USE_MODULES=true`) to prefer `module load` instead
- Generates agent settings even if the CLIs are not currently installed or loaded
- Uses unrestricted defaults (`bypassPermissions` for Claude, `never`/`none` for Codex) ‚Äî same as non-CHPC
- Symlinks `chpc/CLAUDE.md` ‚Üí `~/CLAUDE.md` (the Notchpeak HPC agent guide every CHPC agent loads), via `link_chpc_agent_guide`. This is the only file deployed to `~/CLAUDE.md` on CHPC (on CloudLab, `cloudlab/CLAUDE.md` deploys there instead); edit `chpc/CLAUDE.md`, never `~/CLAUDE.md` directly. Skipped off-CHPC.

**Module name candidates** (used with `--use-modules`) ‚Äî verified on Notchpeak (2026-05). Re-verify after CHPC adds/removes/renames modules with `./install.sh --probe-modules`:

```bash
CLAUDE_MODULE_CANDIDATES=("claude")             # claude/2.1.83
CODEX_MODULE_CANDIDATES=("codex")               # codex/0.114.0
GH_MODULE_CANDIDATES=("gh")                      # gh/2.0.0
NODE_MODULE_CANDIDATES=("nodejs")                 # nodejs/22.4.0 (also: node-js/22.4.0)
UV_MODULE_CANDIDATES=("uv")                       # uv/0.10.9
BTOP_MODULE_CANDIDATES=("btop")                   # btop/1.3.2
NVIM_MODULE_CANDIDATES=("nvim/0.11.2" "nvim")     # nvim/0.7.2, nvim/0.11.2 (D)
TREE_SITTER_MODULE_CANDIDATES=()                  # no module ‚Äî falls back to binary/cargo
```

CHPC uses the same repo `ai/claude_settings.json` / `ai/codex_config.toml` as other hosts; only version-adaptive compat files (tmux/vim/gitconfig/bashrc) land in `~/.dotfiles-generated/`.

**`scripts/chpc-allocs.py`** requires Python 3.7+ (it raises `SystemExit` on older interpreters). Some CHPC nodes ship `/usr/bin/python3` as 3.6 ‚Äî `module load python/3.10.3` before running, or rely on the wrapper installed in `~/.local/bin` to find a usable interpreter.

## CloudLab

`is_cloudlab()` in `lib/common.sh` detects CloudLab/Emulab nodes via hostname (`*.cloudlab.us`, `*.emulab.net` ‚Äî which also covers `*.apt.emulab.net`) or the co-presence of `/usr/testbed` and `/var/emulab` (the reliable signal, since the short hostname doesn't carry the domain). Set `CLOUDLAB=1` to force it. When on CloudLab, `install.sh` symlinks `cloudlab/CLAUDE.md` ‚Üí `~/CLAUDE.md` via `link_cloudlab_agent_guide` ‚Äî the bare-metal agent guide every CloudLab agent loads. Edit `cloudlab/CLAUDE.md`, never `~/CLAUDE.md` directly. Skipped off-CloudLab.

`is_chpc()` and `is_cloudlab()` never both match on a real host ‚Äî their hostname patterns and path markers are disjoint ‚Äî so although both `link_chpc_agent_guide` and `link_cloudlab_agent_guide` target `~/CLAUDE.md`, only one links (the chpc step runs first; if both env overrides `CHPC=1`/`CLOUDLAB=1` were set, cloudlab would win). Unlike CHPC (which hardcodes the user's account/identity), the CloudLab guide is identity-discoverable: experiments are ephemeral, so it tells the agent how to find its project/node/cluster at runtime (`geni-get`, `/proj/*`, `hostname`) rather than baking anything in.

## Do Not

- **Do not** add plugins to `editor/vimrc` ‚Äî it is intentionally plugin-free.
- **Do not** edit files in `~/.dotfiles-generated/` ‚Äî they are overwritten by `install.sh`.
- **Do not** hardcode absolute paths in scripts ‚Äî use `$DIR`.
- **Do not** use `set -e` in scripts ‚Äî `run_step` handles errors.
- **Do not** commit auth files ‚Äî see `.gitignore`.
- **Do not** unload `chpc/1.0` if running on CHPC ‚Äî it is a sticky module required for the environment.
- **Do not** edit `~/CLAUDE.md` directly on CHPC or CloudLab ‚Äî it is a symlink; edit the source guide in `chpc/` or `cloudlab/`.

## Lint / pre-commit

A pre-commit hook at `.githooks/pre-commit` runs `bash -n` on staged shell files, runs warning-level `shellcheck` if available, and scans staged diffs for common secret patterns. It is Bash 3.2-compatible for macOS control hosts. `install.sh` wires this via `git config core.hooksPath .githooks` on first run.
